# ABI Stability

This document explains how the custom C ABI is defined, version-tracked, and
updated in rusted-geom.

---

## What is the ABI baseline?

`abi/baseline/rgm_abi.json` is a snapshot of the public C ABI surface:

- Every `#[no_mangle] pub extern "C"` function annotated with `#[rgm_export]`
- Every `#[rgm_ffi_type]`-annotated type (enums, structs, transparent wrappers)

The file is generated by the `tools/abi-gen` binary and committed to version
control so that CI can detect accidental breaking changes.

**Contents of the file:**
- `"crate_name"` — always `"kernel"`
- `"version"` — the crate version from `Cargo.toml`
- `"abi_hash"` — a SHA-256 hash of the canonical sorted function/type list
- `"functions"` — exported function signatures (Rust name, C name, TypeScript
  receiver/name, parameter types, return type)

---

## What the ABI hash covers

The hash is computed over the sorted, serialised list of all exported function
signatures and FFI type definitions.  It changes when any of the following
change:

- A function is added, removed, or renamed (`rust_name` or `c_name`)
- A function parameter type, name, or order changes
- A return type changes
- A `#[rgm_ffi_type]` struct or enum field name, type, or order changes
- A `#[rgm_ffi_type]` enum variant name or discriminant value changes

The hash does **not** change for purely internal changes (new private functions,
doc comments, module restructuring that does not alter exported symbols).

---

## Breaking vs. additive changes

| Change | Impact | Required action |
|--------|--------|-----------------|
| Remove an exported function | **Breaking** | Major version bump, update TypeScript bindings |
| Rename a function (C name) | **Breaking** | Major version bump |
| Change a parameter type | **Breaking** | Major version bump |
| Add a new exported function | Additive | Minor version bump, regenerate TypeScript bindings |
| Add a new `#[rgm_ffi_type]` struct field at the end | **Breaking** (changes struct layout) | Major version bump |
| Change a private/internal function | Non-breaking | No action needed |
| Add/modify doc comments | Non-breaking | No action needed |

The TypeScript runtime in `bindings/web/src/generated/native.ts` is generated
from the ABI snapshot and must be regenerated after any ABI change.

---

## How to regenerate the baseline

After making intentional ABI changes, update the baseline with:

```bash
scripts/update_abi_baseline.sh
```

This runs `tools/abi-gen`, writes the new `abi/baseline/rgm_abi.json`, and
regenerates `bindings/web/src/generated/native.ts`.

Verify the hash changed for the expected reasons before committing.

---

## The `#[rgm_export]` macro

The `#[rgm_export]` attribute (from `crates/kernel-abi-meta`) marks a
`#[no_mangle] pub extern "C"` function for inclusion in the ABI snapshot.  It
accepts two optional arguments:

```rust
#[rgm_export(ts = "publicTsName", receiver = "receiverName")]
```

| Argument | Meaning | Example |
|----------|---------|---------|
| `ts = "..."` | Public TypeScript method name on the receiver object | `ts = "pointAt"` |
| `receiver = "..."` | Which TypeScript object exposes this method | `receiver = "curve"`, `receiver = "kernel"`, `receiver = "kernel_static"` |

If `ts` is omitted the function is exported at the interop layer only (not
surfaced as a named method in the TypeScript session API).

If `receiver` is `"kernel_static"` the method is on the `KernelRuntime` object
(factory / lifecycle).  All other receivers are on `KernelSession` sub-clients.

---

## CI check

The CI pipeline runs `scripts/check_abi_compat.sh` which:

1. Regenerates the ABI snapshot from the current source
2. Compares the `"abi_hash"` against `abi/baseline/rgm_abi.json`
3. Fails the build if they differ

To **intentionally** update the ABI:

1. Make your changes to the Rust source
2. Run `scripts/update_abi_baseline.sh`
3. Commit the updated `abi/baseline/rgm_abi.json` along with your source changes
4. Describe the ABI change in the commit message and `CHANGELOG.md`

## Managed Binding Readiness (.NET)

The same ABI rules are used for future managed bindings (starting with .NET):

- Keep `rgm_*` symbol names stable unless a major ABI version bump is planned.
- Keep status-first APIs (`RgmStatus` return + out pointers).
- Keep handles opaque and stable in size/layout.
- Preserve two-pass variable-length copy conventions.
- Preserve UTF-8 message conventions for diagnostics.

Breaking ABI changes for managed bindings include:

- changing enum discriminants
- changing `#[repr(C)]` field order/types
- changing pointer ownership contracts
- changing handle widths or sentinel semantics
